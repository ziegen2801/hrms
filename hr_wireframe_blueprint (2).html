1.Purpose and Method

This document turns a client requirements meeting into a buildable specification. The client described their current Darwinbox usage in enough detail to reveal not just which modules they use, but specifically where the tool breaks down for them: offer letters that can't be generated from interview data, a help desk whose email intake misroutes tickets, and an attendance regularization window that needs a hard cutoff. Those specifics are treated as requirements, not incidental complaints.

Alongside the client's notes, we independently analyzed two reference products: Darwinbox, the platform the client currently runs, and Zoho People Plus, a differently structured suite that bundles a comparable set of HR functions from a more modular, configuration-first starting point. The comparison in Section 2 is there to make one thing explicit: we are not rebuilding Darwinbox's screens from memory, we are building the modules the client actually uses, shaped by where Darwinbox causes friction and where Zoho People Plus's structure handles the same job more directly.

The rest of this document is the blueprint a designer needs to start wireframing in Figma without re-asking these questions: the full module tree (Section 5), a screen-by-screen specification for every module (Sections 6 through 20), the workflows that cross module boundaries (Section 21), and how to organize the actual Figma file (Section 22).

2.Reference Analysis: Darwinbox and Zoho People Plus

Darwinbox is an enterprise-grade, end-to-end HCM platform: Core HR, recruitment and onboarding, payroll, leave and attendance, performance, and analytics, all on one data model, built for large enterprises with complex, multi-entity structures. Independent reviews consistently describe it as comprehensive but configuration-heavy: workflows are powerful but rigid once set up, and changes to process or reporting often need admin or vendor support rather than being something an HR user configures directly. That rigidity is exactly what the client's own account of Darwinbox describes: an offer letter process that can't connect directly to interview data, and a help desk whose escalation and routing logic doesn't hold up when tickets arrive by email.

Zoho People Plus takes a different shape: a suite of separately named but tightly integrated applications, Zoho People for core HR and time and attendance, Zoho Recruit for talent acquisition, Zoho Expense, Zoho Payroll, and Zoho Connect for engagement, unified under one login and one employee record. Two things about it are directly relevant here. First, its help desk module (renamed from "Cases" to "HR Help Desk") is built specifically for internal HR queries, with a category structure and a manager-facing escalation path, the same shape the client is asking for, rather than a generic IT ticketing tool repurposed for HR. Second, its configuration model leans no-code: category admins, module admins, and workflow rules are things an HR administrator sets up directly, not things that require a support ticket to change.

What we are taking from eachWhy
Darwinbox's end-to-end scope: recruitment through exit on one employee recordThis is the shape the client already operates in and expects; nothing about their current process needs to be reinvented, only fixed where it breaks.
Zoho People Plus's category-based, no-code HR help desk structureIt is the closest existing match to what the client described: an HR-only service desk with configurable categories, ownership and escalation, not a generic ticketing tool.
Zoho People Plus's admin-configurable workflow and policy modelThe client's complaints about Darwinbox are largely configuration rigidity; a platform where HR administrators can adjust policies and escalation rules directly avoids repeating that problem.
Neither product's exact screen layoutWe are building the client's process, informed by both products' structure, not cloning either one's interface.

4.Role and Access Model

Four roles cover everything described in the client's notes. Every screen specification from Section 6 onward states which of these roles can see it.

RoleScope
Super AdminFull platform configuration: role assignment, module enablement, escalation matrix, policy library, integrations. Typically one or two people per client organization.
Admin (HR Admin)Day-to-day HR operations: employee records, recruitment, help desk ownership and category configuration, document generation, reports. Does not configure platform-wide settings reserved for Super Admin.
ManagerA subset of Admin scoped to their own reporting line: approves their team's leave and regularization requests, views their team's profiles and documents, can raise help desk tickets on a report's behalf.
EmployeeSelf-service only: own profile, attendance, leave, help desk tickets, and document requests.

A fifth, non-authenticated actor, the Candidate, uses only the public career page and candidate registry (Section 7) and never sees the internal platform.

Proxy login. Admin and Super Admin can select any employee and view the platform exactly as that employee would see it, same navigation, same data scope, same permissions, with a persistent banner identifying whose view is being shown and a one-click exit back to the admin's own session. This is a view-only impersonation for support and troubleshooting, not a separate role; it borrows the selected employee's exact permission set for the duration of the session.

22.Figma File Structure and Design System Guidance

The module tree in Section 5 maps directly onto Figma pages, so the designer isn't inventing an information architecture from scratch, only wireframing what's already specified.

Page organization

One Figma page per module number in Section 5 (fifteen pages total), plus two structural pages at the top of the file: 00 – Design System (color, type, spacing, and component tokens) and 01 – Flows (the three cross-module diagrams from Section 21, wired as connected frames using Figma's own flow-arrow feature, not static images).

Frame naming

Each screen becomes one frame, named to match this document exactly, for example 06.3 Regularization request, so a reviewer can trace any frame back to its specification line without guessing.

Component library

Build shared components once and reuse them across every module rather than redrawing per screen: the approval-chain widget (used by leave, regularization, offers and expenses), the SLA countdown badge (help desk), the status pill (active/inactive/pending, ticket status, offer status), and the policy-layer indicator (organization/manager/individual override).

States to design per screen

At minimum: empty state, populated state, and the specific edge case called out in that screen's specification, the five-day regularization cutoff, the SLA-breach countdown turning red, the QR verification page's "revoked" state. These edge cases are the actual requirements; a wireframe that only shows the happy path hasn't captured what the client asked for.

Responsive scope. Employee-facing screens (profile, attendance punch, leave apply, help desk, document requests) need a mobile frame alongside desktop, since punch-in and leave requests are mobile-first actions for most employees. Admin and configuration screens (policy engine, reports builder, escalation configuration) can be desktop-only for this phase.
